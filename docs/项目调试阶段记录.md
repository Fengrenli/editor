# Pascal / Reditor 编辑器 — 项目调试阶段记录

本文档按时间线梳理二次开发与调试中的**主要阶段**：每阶段包含目标、关键改动位置与可复现的操作要点，便于日后回顾、交接或继续迭代（如汉化、安装包封装）。

> **仓库结构提示**：可编辑 UI 与业务主要在 `packages/editor`；Next 壳应用在 `apps/editor`。

---

## 阶段 1：多项目与本地会话（新建 / 切换项目）

**目标：** 支持多个本地工程、独立场景存储与切换，避免单 key 覆盖。

**关键路径（示例）：**

- `packages/editor/src/lib/local-projects.ts` — 本地项目列表与当前项目 ID。
- `packages/editor/src/lib/scene.ts` — 场景持久化 key（如 `pascal-editor-scene:<id>`）、旧数据迁移。
- `packages/editor/src/hooks/use-auto-save.ts`、`use-local-editor-session.ts` — 自动保存与会话就绪状态。
- `packages/editor/src/components/ui/sidebar/panels/settings-panel/` — Projects 等与项目相关的 UI。
- `apps/editor/app/page.tsx` — 入口与 `Editor` 挂载参数。

**操作与验证要点：**

- 在设置侧栏中创建 / 切换项目，确认切换前会 flush 保存、切换后场景与选择状态正确。
- 确认旧版单项目数据能迁移到新 key 方案（若仍存在 legacy 数据）。

---

## 阶段 2：3D / 2D 视角与原点显示

**目标：** 在三维视图与平面图中一致地标示世界原点方向（如 +X / +Z），便于对位与理解轴向；轴线需在深色场景与网格背景上足够醒目。

**关键路径：**

- `packages/editor/src/components/editor/origin-axes.tsx` — 3D 世界原点轴（+X 红、+Y 绿、+Z 蓝），随当前 Level 的 Y 与网格对齐，仅编辑器层。
- `packages/editor/src/components/editor/floorplan-panel.tsx` — 内联组件 `FloorplanOriginAxesLayer`：2D 平面图过原点的 +X（水平）/ +Z（竖直）轴线与 +X、+Z 标签。

**可见性加强（2026-03-29 迭代）：**

| 视图 | 做法摘要 |
|------|----------|
| 3D | 提高顶点色饱和度；`AXIS_LENGTH` 约 3.8m；`lineBasicMaterial` 使用 `toneMapped={false}`、`depthTest={false}`、`renderOrder={1000}`，避免被色调映射压灰、易被几何遮挡。 |
| 2D | 主线宽约 `0.095`（高于主网格 `0.04`）；轴线色 `#ff1a1a` / `#00e676`；每条轴双层绘制（宽光晕 + 主线，`strokeLinecap="round"`）；标签与轴同色、`fontSize` 约 `0.18`、`fontWeight` 700，并带描边（`paintOrder="stroke fill"`）；光晕颜色随 `isDark` 区分深浅主题。 |

**操作与验证要点：**

- 在 3D 中选中建筑 **Level** 后确认原点三轴可见、方向与右手系一致。
- 打开平面图，确认 2D 原点方向与 3D 约定一致（见阶段 7）；轴线在亮/暗主题下均清晰。
- 已移除的旧实现：原 `origin-lasers.tsx` 一类激光式指示（若仓库中已删除，以当前树为准）。

## 阶段 3：墙体精度与接缝

**目标：** 减少墙体 Drafting / 斜接处的数值误差与视觉接缝（如极小缝隙）。

**关键路径：**

- `packages/editor/src/components/tools/wall/wall-drafting.ts`
- `packages/core/.../wall-mitering.ts`（具体路径以仓库内 `wall-mitering` 为准）
- `packages/ui/panels/wall-panel.tsx` 或 `packages/editor` 内对应墙体属性面板

**操作与验证要点：**

- 绘制斜接墙、微调长度与角度，观察接缝与 0.001m 量级精度相关行为是否符合预期。

---

## 阶段 4：UI 层级（z-index）

**目标：** 平面图、相机提示、底部操作条等叠放顺序正确，互不遮挡或误挡。

**约定（摘要）：**

- 平面图相关：`z-40` 量级。
- 相机操作提示：`z-[45]` 量级。
- 底部 `ActionMenu`：`z-50` 量级。

**操作与验证要点：**

- 同时打开平面图与编辑器提示，确认点击与可见性符合设计。

---

## 阶段 5：本地平面图上传、引用与删除

**目标：** 支持本地上传参考图，正确写入资源存储；删除时能清理 `asset://` 等引用。

**关键路径：**

- `packages/editor/src/lib/local-reference-upload.ts`
- `packages/core/src/lib/asset-storage.ts` — 如 `removeAssetByUrl` 等。
- `Editor` 默认 `onUploadAsset` / `onDeleteAsset` 与 `site-panel` 中删除逻辑联动。

**操作与验证要点：**

- 上传 → 在场景/侧栏中可见；删除条目 → 本地与内存引用不再残留无效 URL。

---

## 阶段 6：删除工具条与尺寸标注遮挡

**目标：** 选中物体时的浮动操作条不要挡住墙长等测量/标注。

**关键路径：**

- `packages/editor/src/components/editor/floating-action-menu.tsx` — 抬高或调整挂载位置。
- `packages/editor/src/components/editor/node-action-menu.tsx` — 仅删除模式时的半透明等样式，降低视觉遮挡。

**操作与验证要点：**

- 选中墙/板，开启尺寸显示，确认删除条不再盖住读数。

---

## 阶段 7：2D 平面图与 3D 坐标对齐

**目标：** 平面图 SVG/屏幕映射与 3D 世界坐标同号、同向，避免左右/上下与 3D 相反。

**关键路径：**

- `packages/editor/src/components/editor/floorplan-panel.tsx` — `toSvgX` / `toSvgY` 等映射（**不取反**或与 3D 一致的处理）。

**操作与验证要点：**

- 同一构件在 3D 与 2D 中的左右、前后是否与直觉一致。

## 阶段 9：next-intl 与 Turbopack 开发模式

**现象：** `next dev`（Turbopack）下 `getLocale()` 报错：`Couldn't find next-intl config file`。

**处理：** 在 `apps/editor/next.config.ts` 的 `turbopack.resolveAlias` 中显式增加：

- `'next-intl/config': './i18n/request.ts'`

**操作与验证要点：**

- 结束旧 `next dev` 进程后重启；访问 `/` 应 200，Icon Rail 为中文（默认 `zh-CN`）。

## 阶段 15：`next-intl` 与 R3F Canvas / `<Html>` 门户（浮动操作条运行时错误）

**背景现象：** 汉化后在浏览器中出现两类报错（栈顶常指向 `NodeActionMenu` / `useTranslations` 或 `useEditorCanvasI18n`）：

1. `NextIntlClientProvider` context 未找到（`useTranslations`）。
2. `useEditorCanvasI18n must be used under EditorCanvasI18nProvider`。

**根因概要：**

| 层级 | 说明 |
|------|------|
| R3F `<Canvas>` | 使用与页面 DOM 不同的 reconciler；根布局里的 `NextIntlClientProvider` **不会**透传到 Canvas 子树，`useTranslations` 在纯 Canvas 子组件内不可用。 |
| `useContextBridge`（`@react-three/drei`） | 必须在 **仍处于 DOM React 树、且已是 `EditorCanvasI18nProvider` 子代** 的组件里调用；若把「仅含 `useContextBridge` 的桥」放在 `<Canvas>` **内部**作为 `Viewer` 的子节点，此时读取 `EditorCanvasI18nContext` 仍为 **null**，桥接不到任何值。 |
| `drei` 的 `<Html>` | 将子内容挂到独立 DOM 入口，内部组件 **不继承** R3F 侧由 Bridge 注入的 context；故 **`NodeActionMenu` 放在 `<Html>` 内时不能直接 `useContext` / `useEditorCanvasI18n`**。 |

**最终方案（代码落点）：**

- `packages/editor/src/contexts/editor-canvas-i18n.tsx`  
  - `EditorCanvasI18nProvider`：在 DOM 树中用 `useTranslations` 组装 `nodeMenu`、`resolveToolLabel` 等，写入 `EditorCanvasI18nContext`。  
  - `EditorViewerWithR3fI18nBridge`：在 **Canvas 外**调用 `useContextBridge(EditorCanvasI18nContext)`，再渲染 `<Viewer><IntlBridge>{children}</IntlBridge></Viewer>`。
- `packages/editor/src/components/editor/index.tsx`：用 `EditorViewerWithR3fI18nBridge` 替代「`Viewer` + 在 Canvas 内套桥」；整体仍包在 `EditorCanvasI18nProvider` 内。
- `packages/editor/src/components/editor/floating-action-menu.tsx`：在 `<Html>` **外**调用 `useEditorCanvasI18n()`，将 `nodeMenu` 以 **`nodeLabels` props** 传入 `NodeActionMenu`。
- `packages/editor/src/components/editor/node-action-menu.tsx`：支持可选 `nodeLabels`；无 props 时回退 `useContext(EditorCanvasI18nContext)`（平面图 DOM 场景）；文案字段使用 `labels.move` 等对象属性（非 `t('key')`）。  
  - 导出 `EditorCanvasI18nContext` 供回退读取。
- `packages/editor/src/components/tools/shared/cursor-sphere.tsx`：继续在 **`<Html>` 外**用 `useEditorCanvasI18n().resolveToolLabel` 计算 `toolAlt`，仅把结果传入 portal 内（避免在 portal 内读 context）。

**推荐调试方法（可复用）：**

1. **看栈**：区分错误出在「布局侧 / R3F 内 / `<Html>` 内」—— portal 内优先怀疑 context 断裂。  
2. **对照 Drei 文档**：`useContextBridge` 必须在 **已挂好上游 Provider** 的 DOM 组件中创建，再把返回的 `Bridge` 放进 `<Canvas>`。  
3. **最小复现**：仅选中物体触发 `FloatingFocusMenu` → `NodeActionMenu`，确认是否与 Html 同现。  
4. **验证命令**：`npx turbo run build --filter=editor`；本地 `next dev` 后选中物体（`FloatingActionMenu`）、打开平面图洞口菜单、搭建模式光标提示，确认无红屏。

**与打包关系：** 上述为纯前端运行时问题，与 `next build` 是否通过无必然联系；修复后桌面壳 / 内嵌 WebView 只要加载同一套页面，行为与浏览器一致，**可进行打包**。

## 阶段 16：Windows 桌面壳（Electron）打包与启动

**目标：** 使用 `apps/desktop` 将 Next standalone 嵌入 Electron；安装包可启动；开发机可跳过重复下载 Electron 二进制。

**关键路径：**

- `apps/desktop/main.cjs` — 以 `ELECTRON_RUN_AS_NODE` 子进程启动 `resources/next/apps/editor/server.js`；监听 `127.0.0.1` 与固定端口（默认 `43120`）；失败时用 `dialog.showErrorBox` 提示（GUI 下无控制台，勿仅 `console.error`）。
- **子进程 stdio：** 不可将尚未获得有效 `fd` 的 `fs.createWriteStream` 直接作为 `spawn` 的 `stdio`（会触发 `ERR_INVALID_ARG_VALUE`）。应使用 `stdio: ['ignore','pipe','pipe']`，再将 `stdout`/`stderr` `pipe` 到日志文件；`app.whenReady().then(...).catch(...)` 避免未处理的 Promise 拒绝。
- `apps/desktop/scripts/prepare-next.cjs` — 从 `apps/editor/.next/standalone` 与 `static`、`public` 拷入 `apps/desktop/dist/next`。在 **Bun** 等布局下，`standalone/apps/editor/node_modules/next` 常为指向仓库外 store 的 **符号链接**；若用 `fs.cpSync` 原样复制，安装包内仍指向本机绝对路径，解压到其它盘或他人电脑会 **`Cannot find module 'next'`**。脚本使用 **`copyTreeResolvingSymlinks`**，遇 symlink 则复制真实目录，目标树中不再含该链接。
- `apps/desktop/electron-builder.config.cjs` — 读取环境变量 **`REDITOR_ELECTRON_DIST`**（已解压的 Electron 根目录，内含 `electron.exe`，或指向 `electron.exe` 的路径），设置后 **electron-builder 使用本地二进制，跳过从 GitHub 下载 zip**。
- `apps/desktop/scripts/ensure-local-electron-env.cjs` + `package.json` 脚本 **`build:desktop:local-electron`** — 打包前校验变量与 `electron.exe` 是否存在。
- `packages/viewer/package.json` — `peerDependencies` 中 `@pascal-app/core` 须与 monorepo 内 core 主版本一致（例如 `^0.3.0`）；若仍为 `^0.1.4`，npm 可能从 registry 解析到旧版 core，与 `three@^0.183` 产生 `ERESOLVE`。
- **产品名与图标（Rdesign）：** `package.json` 的 `build.productName` 为 **`Rdesign`**，`appId` 为 `app.rdesign.desktop`；资源目录 **`apps/desktop/installer/`**（避免与仓库根 `.gitignore` 的 `build` 规则冲突）。脚本 **`prepare-icon.cjs`** 在打包前将源图缩放为 **1024×1024** PNG（白底 `contain`）→ `installer/icon.png`，供 electron-builder 生成 `.ico`。源图优先级：环境变量 `RDESIGN_ICON_SOURCE` → 仓库根 **`LOGO1.jpg` / `LOGO1.png`** → `installer/source-logo.*`。依赖 **`sharp`**：在 **`apps/desktop`** 执行 **`npm install`** 即可；若 npm 报错可试 **`npm install --legacy-peer-deps`**。单独执行：`npm run prepare-icon`。

**与「SWC / asarUnpack」类通用教程的对照（勿照搬）：**

| 说法 | 在本仓库是否成立 |
|------|------------------|
| SWC 是编译器；运行时代码会依赖 **`@swc/helpers`** | **成立**（便于理解，类比 VC++ 运行库也合理）。 |
| 用 **`asarUnpack`** 把 `@swc/helpers` 从 asar 里拆出来 | **不适用**：Next 产物在 **`extraResources` → `resources/next/`**，与 **`app.asar` 并列**，**本就不在 asar 内**；`asarUnpack` 管不到这块。 |
| 再往 **`extraResources` 加 `node_modules/@swc/helpers`**（相对桌面壳） | **易错**：`apps/desktop` 的 **`package.json` 无此依赖**，路径通常不存在；真正需要的是 **`standalone` 里的 `apps/editor/node_modules`**。 |
| **`main.cjs` 把 stdio 改成全 `ignore`** | **不必**：已用 **`pipe` 写日志**，比全忽略更利于排错。 |
| 日志里出现 **Temp** 路径 | **未必与缺 `@swc/helpers` 同一根因**：**Portable** 等会先解压到 Temp；缺包仍以 **`resources/next/.../node_modules`** 是否随 **`prepare-next` + 先构建 editor** 打全为准。 |

**本仓库已采用的「彻底带上 SWC 运行时」做法：** `apps/editor` 依赖 **`@swc/helpers@0.5.15`** + **`outputFileTracingIncludes`**；**`prepare-next`** 解析 **`next` symlink**、**校验/兜底补拷 `@swc/helpers`**；打包用 **根目录 `npm run build:desktop`** 或 **`apps/desktop` 的 `npm run build:desktop:from-root`**，保证 **先 `editor#build` 再拷贝 standalone**。

**产物与使用：**

- **`apps/desktop/release/Rdesign-Setup-<version>.exe`** — NSIS 安装程序（向导安装）。
- **`apps/desktop/release/Rdesign-<version>-Portable.exe`** — **绿色便携包**（自解压式单文件，适合安装器被 SmartScreen/杀毒拦截时试用）。
- **`apps/desktop/release/win-unpacked/`** — 构建输出目录，运行其中 **`Rdesign.exe`**（需整目录保留）。
- **若双击 Setup 无反应或被拦**：见下方「Setup 无法安装」排查。

- 安装后若内置服务未起：查看 **`%AppData%\Rdesign\rdesign-server.log`**（`main.cjs` 中 `app.setName('Rdesign')` 时 userData 目录一般为 **`Rdesign`**）。
- **Setup 无法安装（双击无反应 / 点不了安装）**：勿从压缩包内直接运行——先**解压到本地磁盘**再双击；安装包上**右键 → 属性**，若有 **解除锁定** 请勾选；**Windows 安全中心 / 第三方杀毒** 查看是否隔离；出现 **Microsoft Defender SmartScreen** 时选 **更多信息 → 仍要运行**。未签名的大型 NSIS 安装包在部分环境下会被静默拦截，而 **`win-unpacked\Rdesign.exe`** 反而能启动——属常见现象；对外分发可优先 **`Rdesign-*-Portable.exe`**（单文件）或 **整夹 zip**（见下条）。
- **用 zip 分发 `win-unpacked` 时**：必须把 **整个文件夹内所有文件** 解压到本地目录后再运行 **`Rdesign.exe`**。不可在资源管理器里 **不解压、直接双击压缩包里的 exe**（缺少同目录 dll / 资源会失败或行为异常）。zip 建议用「解压到 `Rdesign\`」这类方式，避免深层路径过长。  
  **完整自测清单 + 给同事的步骤与可复制话术** 见：**`docs/Rdesign-桌面版分发与使用说明.md`**。
- **同事机「内置服务已意外退出 / 退出代码 1」**：弹窗会尽量附带 **日志尾部**；仍须对照完整日志。常见排查：安装 **[Microsoft Visual C++ 2015–2022 可再发行组件（x64）](https://aka.ms/vs/17/release/vc_redist.x64.exe)**；暂时关闭或放行杀软；确认 **43120** 端口未被占用；清理系统环境变量里的 **`NODE_OPTIONS`**（若曾配置 `--inspect` 等）。`main.cjs` 对子进程环境会 **删除继承的 `NODE_OPTIONS`**，并写入 `[bootstrap]` 行便于定位路径与版本。

## 阶段 17：npm 工作流与依赖树一次性对齐（避免「装一次发现一个错」）

**目标：** 用 **npm** 稳定安装与构建时，不把问题留到 `turbo build` / 桌面包才暴露；与历史上 **bun** 混用、半截 lock、残留子包 `node_modules` 等情况划清界限。

**根因摘要（已在本仓库对策化）：**

| 现象 | 常见原因 | 对策 |
|------|----------|------|
| `npm error Invalid Version:`（版本号为空） | 损坏的 **`package-lock.json`**（`packages` 里大量 **`"…/node_modules/foo": {}`**）；或 **仅删根 `node_modules`** 后旧树与 npm 11 dedupe 冲突 | **`npm run reinstall`**；必要时删掉 **`package-lock.json`** 后再 **`npm install`**；**`package-lock.json` 应提交**，勿加入 `.gitignore` |
| `Cannot read properties of null (reading 'matches')` | 根依赖 **`portless`** 触发 npm 11 arborist bug | 根 **`package.json` 已移除 `portless`** |
| `@pascal-app/core:build` 找不到 **`packages/core/node_modules/typescript/bin/tsc`** | **`packages/*/node_modules/.bin`** 仍为 **bun** 生成的 **`tsc.exe` / `tsc.bunx`**，而 **hoist 后 typescript 只在根 `node_modules`** | **`npm run clean:node_modules`**（删各 workspace 的 **`node_modules`**）再 **`npm install`** |
| Turbo 日志里出现 **`bun.exe run build`** | 根 **`packageManager` 曾为 `bun@…`**，Turbo 按该字段选执行器 | 根 **`packageManager` 已改为 `npm@11.8.0`**（与「仅用 npm」文档一致；可按本机 npm 小版本调整） |

**推荐命令（Windows / PowerShell）：**

```powershell
Set-Location D:\Reditor-main
npm run reinstall
npx turbo run build --filter=editor --force
npm run build:desktop
```

**操作与验证要点：**

- 装依赖前若遇 **`EBUSY`**（常见于 **lightningcss** / **tailwind oxide** 的 `.node`），先停 **dev server、Next、Electron**，必要时结束任务管理器里的 **node** 再执行 **`npm run reinstall`**。
- **`apps/desktop/release/`** 已加入 **`.gitignore`**：避免打包产物里的嵌套 **`package.json`** 混入仓库、干扰检索与认知。
- 详细逐步说明与分发话术仍见：**`docs/Rdesign-桌面版分发与使用说明.md`**。

## 阶段 18：封装验收标准与「下次如何稳交付」（三次迭代经验）

**背景（为何「第一次能封上、后面难封上」）**

| 轮次 | 大致情况 | 启示 |
|------|----------|------|
| 第一次 | 功能较少时 **安装包曾成功** | 依赖面窄、standalone 追踪与 monorepo 边界问题暴露少。 |
| 第二次 | 功能与架构加重后 **封装直接失败** | 失败点多在 **Next standalone / 依赖追踪 / workspace**，与「写业务」不同类。 |
| 第三次 | **绿色版 `win-unpacked` 换盘可用**，**Setup 仍不稳定** | **应用与内置服务已闭环**；安装包受 **NSIS 下载、杀软、未签名、本机策略** 影响，**不应与「能不能发版」画等号**。 |

**「保证最后一定可以封装」——在工程上能承诺什么**

- **无法**在无签名、多网络环境下承诺 **Setup.exe 对所有人 100% 双击成功**（现实约束）。  
- **可以**承诺：按下列 **固定顺序与验收标准** 执行后，**一定能得到可对外分发的绿色版**（**整夹 zip**，见 **`docs/Rdesign-桌面版分发与使用说明.md`** 方式 A）。

**发布前强制检查清单（建议复制为 Issue / PR 模板）**

1. **干净依赖（避免 bun/npm 混树）**  
   - 根目录：`npm install`  
   - 桌面工作区（含 **electron postinstall**、**app-builder-bin**）：`npm install --workspace=desktop --foreground-scripts`  
   - 若刚清过依赖或换过包管理器：优先 `npm run reinstall` 再执行上两行。  
2. **先编 editor，再打桌面**（避免旧 standalone 进包）：  
   - `npx turbo run build --filter=editor --force`  
3. **桌面包**（二选一，**绿色版必达**）：  
   - 常规：`npm run build:desktop`（含 NSIS + portable，可能因网络/杀软失败）；  
   - **保底（只出 `win-unpacked`）**：`cd apps/desktop && npm run build:desktop:dir`（少拉 NSIS 工具链，见阶段 17 修订说明）。  
4. **验收（比「本机在工程目录能开」更严）**  
   - 将 **`release\win-unpacked`** **整夹** 复制到 **另一盘符**（或打成 zip 再解压到新目录），仅运行解压后的 **`Rdesign.exe`**；  
   - **通过 = 本次迭代封装合格**；未通过不得合并发布分支。  
5. **Setup / Portable.exe（可选）**  
   - 作为增值产物：成功则一并提供；失败 **不阻塞** 用 zip 发 **`win-unpacked`**。

**下次迭代可加的「硬保险」（按需）**

- **CI**：在干净 runner 上跑上述 1→2→3（至少 `build:desktop:dir`），把 **`win-unpacked` 打 zip 成 artifact**，避免「只有某台开发机打得开」。  
- **版本锁定**：根目录 **`engines.node`** / 团队统一 **Node 主版本**（如 20 LTS），减少「我机 npm 11 你机不同」的差异。  
- **安装包纳入硬指标的时机**：有 **代码签名证书** 与稳定 **electron-builder 二进制镜像**（或内网缓存）之后，再把 **Setup 成功率** 与绿色版并列。
